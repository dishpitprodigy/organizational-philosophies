# Context: Managed Runoff for Deprecated Processes

## Purpose

Use this context to update the organizational philosophy documents with a clearer framework for handling deprecated processes, systems, designs, or operational models.

The core idea: once an organization has decided that a process/design/system is no longer the desired future state, it should stop treating that thing as an investment object. It should enter **managed runoff**: a bounded transition state where the goal is not improvement, but safe, controlled exit.

## Core Thesis

Once a process, platform, design, or service has been deprecated, it should no longer receive normal improvement investment.

Work on the deprecated thing should be limited to:

* keeping it alive long enough to exit safely
* preventing material security, compliance, or safety issues
* preserving availability during the transition
* enabling migration to the replacement path
* extracting knowledge that would otherwise be lost
* reducing risk associated with shutdown or cutover

Anything beyond that should require explicit justification.

The default assumption should flip:

> Before deprecation, improvement work may be investment.
> After deprecation, improvement work is presumed waste unless it directly reduces exit risk or accelerates retirement.

## Important Distinction

This is related to, but not the same as, the **Strangler Fig pattern**.

Strangler Fig answers:

> How do we incrementally move functionality away from the old system/process?

Managed runoff answers:

> How should the organization treat the thing being left behind?

Strangler Fig is a migration pattern. Managed runoff is a governance / lifecycle / portfolio-management concept.

## KO vs Kill Distinction

A useful metaphor:

### KO

The old process has been neutralized as the primary path.

It may still exist as:

* fallback
* read-only reference
* emergency path
* support path for unmigrated consumers
* temporary compatibility layer

A KO is not the same as removal. It means the old thing is no longer the strategic operating model.

### Kill

The old process is fully removed from the organization’s operating model.

It has:

* no active consumers
* no normal support obligation
* no enhancement path
* no hidden dependencies
* no planning allocation
* no privileged exception status
* no reason to keep appearing in roadmap discussions

The kill state is the actual retirement target.

A common organizational failure is mistaking KO for kill. The old thing stops being the main path, but still survives indefinitely as a zombie process.

## Zombie Process Failure Mode

A zombie process is a deprecated process that continues to consume labor, planning attention, maintenance effort, and political oxygen even after the organization has decided it should not be the future state.

Zombie processes survive because no one forces a clear lifecycle decision.

Typical symptoms:

* “We know this is going away, but…”
* “We just need to clean this up first.”
* “We should document the old process better.”
* “We need to make this more robust while we migrate.”
* “We can’t remove it yet because someone might still need it.”
* “Let’s add this one feature to the legacy path.”
* “We’ll revisit this next quarter.”

Most of those may sound responsible, but they often disguise the same problem: the deprecated thing is still being treated like an asset instead of a liability under managed runoff.

## Better Framing

Once something has been deprecated, the organization should explicitly classify work against it.

Useful categories:

### Allowed Work

Work is allowed when it directly supports:

* safety
* security
* compliance
* stability during transition
* migration
* dependency discovery
* knowledge extraction
* shutdown readiness
* emergency repair

### Questionable Work

Work should be challenged when it involves:

* polishing the deprecated path
* improving user experience on the deprecated path
* expanding old functionality
* adding new integrations
* improving performance beyond transition needs
* refactoring internals without migration benefit
* creating extensive new documentation for a soon-to-die process

### Disallowed by Default

Work should generally be rejected when it:

* increases attachment to the deprecated process
* creates new consumers
* extends the retirement timeline without explicit approval
* makes the deprecated path more attractive than the replacement
* consumes capacity needed for the target-state work
* preserves ambiguity about the future state

## Key Principle

The deprecated system/process should be maintained only enough to exit.

Not enough to make it good.

Not enough to make it elegant.

Not enough to make people comfortable staying there.

The job is to reduce carrying cost and retire the thing safely.

## Useful Phrase

> Once a design or process has been formally deprecated, it exits the investment portfolio and enters managed runoff. Work on it should be limited to safety, continuity, compliance, migration enablement, and knowledge extraction. Any improvement beyond that requires explicit justification because the default value classification has changed from investment to carrying cost.

## Related Frameworks / Vocabulary

This idea overlaps with several existing frameworks:

* **ITIL service transition / service retirement**
* **Application Portfolio Management**
* **Gartner TIME model: Tolerate, Invest, Migrate, Eliminate**
* **Enterprise Architecture migration planning**
* **Technical debt management**
* **Legacy-system decommissioning**
* **Strangler Fig migration pattern**
* **Runoff / rundown concepts from finance and operations**

The useful move is to combine them into plain operational language:

> Is this still an investment asset, or is it now a managed runoff liability?

## Planning Implications

For any deprecated process/system, the organization should define:

1. **Deprecation date**
   When did we decide this is no longer the desired future state?

2. **Replacement path**
   What is the target process/system/design?

3. **Runoff owner**
   Who owns keeping the old thing safe until exit?

4. **Retirement owner**
   Who owns making the old thing disappear?

5. **Allowed maintenance scope**
   What work is still justified?

6. **Forbidden maintenance scope**
   What work should no longer be accepted?

7. **Exit criteria**
   What must be true before the old thing can be killed?

8. **Kill date or review date**
   When does the organization force the next decision?

9. **Exception policy**
   What kinds of security, safety, compliance, or operational issues justify touching the old path?

10. **Dependency inventory**
    Who or what still depends on the old process?

## Argument Against Over-Maintaining Deprecated Processes

Maintaining deprecated systems is not automatically responsible.

It can be irresponsible when it:

* steals capacity from the replacement
* delays migration
* conceals the true cost of the old process
* normalizes indefinite transition states
* lets stakeholders avoid making hard decisions
* allows old process defects to keep shaping the future design
* creates the illusion of progress while preserving the same failure mode

The responsible act is not always “maintain the old thing better.”

Sometimes the responsible act is:

> stop investing, contain the risk, migrate the consumers, and kill it.

## Anti-Pattern: Replacement Without Runoff Discipline

Organizations often approve a new process or platform but fail to define the lifecycle of the thing it replaces.

That creates two simultaneous operating models:

* the old model, still unofficially alive
* the new model, officially strategic but capacity-starved

This creates duplicated work, unclear ownership, inconsistent standards, and endless exceptions.

A replacement strategy without runoff discipline is not a full strategy. It is just additive complexity.

## Tone / Style Guidance

Keep the language direct and operational.

Avoid making this sound like a purely theoretical framework. The point is practical: organizations waste enormous effort maintaining things they already decided to abandon.

Use language like:

* managed runoff
* carrying cost
* deprecated path
* target state
* exit criteria
* retirement owner
* investment object
* liability under transition
* migration enablement
* zombie process
* kill criteria
* exception-only maintenance

Avoid language that makes deprecated systems sound like they are still normal platforms deserving full product treatment.

## Compact Summary

Managed runoff is the discipline of treating deprecated systems, processes, or designs as controlled-exit liabilities rather than investment assets.

The goal is not to make the old thing better. The goal is to keep it safe enough, stable enough, and understandable enough to remove it without creating unacceptable risk.

A deprecated process should not receive normal improvement investment. It should receive only the work required to sustain, migrate, contain, and retire it.
